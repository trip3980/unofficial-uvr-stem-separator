# OpenStem Workflow Audit

Status boundary: Hardened Functional Alpha. One local CPU separator proof lane has passed with a verified model and real non-empty output stems. Beta Candidate still requires final release checklist review, packaged-app review, and user approval.

## Is The Programming Language Causing The Problem?

The issue is probably not React, TypeScript, or Electron themselves. The issue is likely that the implementation allowed planned, mocked, browser-only, and verified states to coexist without one shared truth model. The fix is shared state discipline and UVR5-style workflow simplicity, not changing programming languages.

## Root Cause

Contributing causes:

- Bad state architecture: each panel owns its own readiness language, so truth states drift.
- Fragmented component logic: Audio Separator, Model Manager, Mixer, Ensemble Planner, and Generative Lab each restate backend/proof status differently.
- Missing shared status model: the app needs one small vocabulary for `not_selected`, `preview_only`, `not_checked`, `blocked`, `verified`, `running`, `failed`, and `completed`.
- AI-generated placeholder behavior: planned panels and optimistic defaults were added before the local backend proof path was proven.
- Overcomplicated UI copy: technical explanations appear in creator-facing screens where a simple next action would work better.
- Disconnected backend contract: the native bridge is real, but UI components still infer readiness locally instead of consuming one authoritative run-readiness object.
- Lack of UVR5 workflow discipline: advanced panels were allowed to sit beside the core path before the core path became visually dominant.
- Genuine technical limitation: only one local CPU proof lane is proven. GPU, cross-platform, full catalog, bundled model distribution, and professional stem-quality claims remain unproven and should not be hidden.

## Restoration Plan

1. Define a shared `RunReadinessState` used by Audio Separator, Model Manager, Mixer, and release panels.
2. Make the Audio Separator main screen follow one visible lane: input, output, model, readiness, run, progress, outputs.
3. Move detailed diagnostics behind expandable panels and keep the first screen task-focused.
4. Remove demo/default-looking paths from the active run lane unless they are explicitly labeled preview-only.
5. Keep model verification strict, but present one user path for proof-eligible manual import.
6. Keep planned tools, but group them as secondary until their backend contracts exist.
7. Preserve independent OpenStem naming and attribution. Use UVR-style only as a workflow reference, not as branding or official affiliation.

Principle: When in doubt, prefer the classic UVR5 user path: Select input -> select output -> choose model -> check readiness -> run -> show progress -> show real outputs.

Only add advanced panels after the basic path remains clear.
