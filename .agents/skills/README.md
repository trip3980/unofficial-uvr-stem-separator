# OpenStem Repo-Scoped Skills

These skills narrow Codex behavior for focused OpenStem AI Audio Workstation tasks. They do not replace the root `AGENTS.md`.

- `openstem-release-gate`: release readiness, proof boundaries, Beta gate, final status.
- `model-integrity`: model registry, SHA-256, manual import, proof eligibility, broken links.
- `electron-packaging`: Electron, electron-builder, Windows/Linux/macOS packaging, runtime diagnostics.
- `ai-backend-proof`: Python, FFmpeg, PyTorch, audio-separator, CPU proof, subprocess execution, output verification.
- `browser-vs-native-truth`: browser preview vs Electron native execution boundaries.
- `missing-resource-recovery`: Use when blocked, missing, broken, auth-required, not-checked, or unavailable resources need diagnosis and recovery. This skill turns vague blocked states into structured recovery paths without weakening proof or model integrity.
- `react-electron-ipc`: preload/main/renderer contracts, IPC, safe filesystem operations.
- `audio-ui-workflow`: Classic Console, Mixer, Ensemble, Batch Encoder, Basic Pitch, Generative Lab, settings UI.
- `third-party-licenses`: README, About, THIRD_PARTY_NOTICES, attribution, license classification.
- `test-and-build-verification`: validation commands and final verification.
- `cross-platform-release`: Linux/macOS release planning and verification.
- `update-workflow`: program updates, app-update manifests, model-catalog refreshes, model-weight replacement, release feeds, installer verification, and updater UI wording. Keeps app updates, catalog metadata, and model weights separate until each lane has real manifest and SHA-256 verification.
- `final-polish-and-handoff`: final cleanup, release handoff, packaging review, documentation closure, blocker list.
- `music-midi-workflow`: Basic Pitch, MIDI export, YuE, Suno, generative music lab, and music connector workflows. Keeps MIDI/generation features honest, separates browser preview from native execution, verifies real local output files, and prevents MIDI/generation from satisfying stem-separation proof.
- `audio-format-workflow`: Batch Encoder, FFmpeg format conversion, codec settings, output formats, sample rate/channel/bit-depth controls, export verification, and audio file validation. Keeps format conversion honest, verifies real output files, separates codec/runtime readiness from AI proof, and prevents conversion tools from satisfying stem-separation proof.
- `mastering-workflow`: Use when modifying Mastering Lab, Web Audio mastering, loudness normalization, reference matching, final audio export, before/after reports, or mastering workflow UX. Keeps mastering local-first, verified, non-destructive, and separate from AI stem-separation proof.
- `audacity-reference-workflow`: Use when adapting Audacity-style patterns for recording, import/export, effect chains, macros, audio analysis, mastering workflows, FFmpeg integration, and recoverable audio editing. Keeps Audacity as a reference while avoiding GPL/code-copy risks and preserving OpenStem's proof boundaries.
- `local-transcription-workflow`: Use when modifying the Local Transcription workspace, Whisper model management, timestamp handling, diarization, PDF/DOCX/TXT/SRT/VTT export, directory transcription, transcript history, or TurboScribe-style local dashboard behavior. Keeps transcription local, stable, non-destructive, and separate from AI stem-separation proof.
- `transcription-intake-workflow`: Use when modifying recording, microphone input, imported audio sessions, VTT/Zoom transcript import, speaker renaming, transcript file renaming, archive/export folders, one-click export, and local transcription intake. Keeps the workflow local, non-destructive, user-friendly, and separate from AI stem-separation proof.
- `transcription-automation-workflow`: Use when modifying end-to-end transcription intake automation, VTT import automation, recording completion flows, archive/export automation, post-processing edits, speaker rename, resave/overwrite behavior, and manual/automatic workflow modes.
- `checkpoint-automation-workflow`: Use when modifying stage-based automation, workflow checkpoints, start/stop automation controls, resume behavior, manual review points, rerun-step behavior, and transcription-to-export-to-prompt pipelines.
- `clinical-prompt-workflow`: Use when modifying Clinical Workflow Builder, transcript-to-clinical-prompt scaffolds, curated clinical local/chat model lanes, local LLM readiness, section outputs, unified EHR text boxes, clinician-review gates, clinical export/history, or draft-only prompt workflow behavior. Keeps clinical drafting separate from proof and blocked until real local model execution exists.
- `clinical-privacy-security`: Use when modifying HIPAA-aware privacy/security language, clinical transcript storage, PHI handling, local-first safeguards, cloud model gates, BAA-required wording, audit logs, clinical risk registers, or privacy-sensitive export/history behavior. Prevents compliance overclaims and keeps cloud PHI processing disabled by default.
- `subq-rag-transcript-workflow`: Use when modifying sub-question decomposition, RAG/evidence retrieval, transcript context maps, prompt-section planning, long-transcript workflows, and local evidence-backed answer generation. Keeps long transcript answers grounded, staged, line-break formatted, and separate from AI stem-separation proof.
- `prompt-library-workflow`: Use when modifying saved prompt templates, workflow presets, prompt section editing, auto-save, template duplication, template import/export, and prompt library UX inside the Transcript Workflow Builder. Keeps prompt reuse seamless, local-first, plain-text oriented, and separate from AI stem-separation proof.
- `document-format-workflow`: Use when modifying transcript/document import, PDF/DOCX/ODT/RTF/TXT/HTML export, VTT-to-document conversion, prompt-output export, document parser dependencies, office-format workflows, or archive/export behavior. Keeps document handling local, flexible, verified, and separate from AI stem-separation proof.
- `voicebox-reference-workflow`: Use when adapting Voicebox-style local voice I/O patterns, capture history, STT model selection, local LLM refinement, queue/retry/recovery, recording UX, post-processing presets, and local-first workflow design. Keeps Voicebox as a reference while preserving OpenStem's proof, privacy, and branding boundaries.

`verified-toolchain-status` is covered inside `test-and-build-verification` and `final-polish-and-handoff`: `release:check`, `proof:check`, strict TypeScript ratchet, artifact verification, and model-source audit remain release-readiness evidence, not AI proof.
